# Feature: Migration Resource Cleanup

## Goal

Implement a new **Migration Resource Cleanup** feature that safely removes resources created by the migration tool from the destination FHIR server.

This feature should reuse as much of the existing migration infrastructure as possible.

---

# Resource Ownership

Only delete resources created by the migration tool.

A resource belongs to the migration tool when its meta extension contains:

* initiator-component = `fhir-migration-tool`

Resources created by other systems must never be deleted.

---

# Cleanup Filters

Support the following filters.

## Resource Types

Reuse the existing resource selection UI (checkboxes).

The user can choose which resource types should be deleted.

---

## Initiator

Only delete resources where:

* initiator-component = `fhir-migration-tool`

---

## Creation Date Range

Allow filtering using:

* From date/time
* To date/time

Use FHIR `_created` search parameters whenever possible.

---

# Cleanup Order

Cleanup must execute using the reverse migration dependency order to minimize referential integrity violations.

Example:

AuditEvent

↓

Consent

↓

Composition

↓

MedicationDispense

↓

MedicationRequest

↓

Procedure

↓

ProcedureRequest

↓

ClinicalImpression

↓

Observation

↓

AllergyIntolerance

↓

Encounter

↓

Condition

↓

Appointment

↓

Slot

↓

Schedule

↓

Coverage

↓

Patient

The dependency order should remain configurable.

---

# Cleanup Process

For each selected resource type:

1. Search matching resources using:

   * initiator-component
   * creation date range

2. Retrieve matching resources using paging.

3. Build FHIR Transaction Bundles containing DELETE operations.

4. Send each Transaction Bundle to the destination FHIR server.

---

# Shared Transaction Bundle Builder

Introduce (or refactor) a shared Transaction Bundle Builder.

This component must be reused by:

* Migration (POST Transaction Bundle)
* Cleanup (DELETE Transaction Bundle)

The application should have only one implementation responsible for:

* building Transaction Bundles
* splitting bundles
* calculating serialized payload size
* enforcing bundle limits

Avoid duplicating bundle-building logic.

---

# Global Transaction Bundle Settings

Move bundle limits into the global application Settings.

These settings must be shared by every feature that sends Transaction Bundles.

Settings:

* Maximum Resources per Transaction Bundle
* Maximum Transaction Bundle Request Size (MB)

Default values:

* Maximum Resources per Transaction Bundle = **100**
* Maximum Transaction Bundle Request Size = **3 MB**

Users should be able to modify these values from the Settings UI without changing source code.

---

# Bundle Splitting Rules

A Transaction Bundle must be finalized whenever either limit is reached:

* Maximum Resources per Transaction Bundle
* Maximum Transaction Bundle Request Size

The request size must be calculated using the serialized JSON payload that will actually be sent.

Do not estimate bundle size using object count.

Reuse the same bundle splitting algorithm for both migration and cleanup.

---

# Progress Reporting

Display progress for every resource type.

Example:

Patient

250 / 250 deleted

Composition

980 / 980 deleted

Observation

3,500 / 3,500 deleted

Also display an overall cleanup progress indicator and a final summary.

---

# Dry Run

Support a dry-run mode.

Dry-run should:

* search matching resources
* count matching resources
* estimate the number of Transaction Bundles

Do not execute any DELETE operations.

---

# Confirmation

Before cleanup begins, display:

* selected resource types
* creation date range
* estimated matching resources
* estimated Transaction Bundle count

Require explicit confirmation before starting deletion.

---

# Error Handling

If a Transaction Bundle fails:

* log the bundle number
* log every resource contained in the bundle
* log the server response

Continue processing remaining bundles whenever possible.

Display failed bundles in the final summary.

---

# Architecture

Implement Cleanup as a separate module while reusing existing infrastructure.

Suggested components:

* Cleanup Service
* Cleanup Searcher
* Cleanup Executor
* Cleanup Progress Tracker

Shared infrastructure:

* FHIR Client
* Transaction Bundle Builder
* Paging
* Settings
* Logging
* Progress Reporting

The shared Transaction Bundle Builder should become the single source of truth for every Transaction Bundle generated by the application.

---

# Expected Result

The user should be able to safely clean up migrated resources by selecting:

* resource types
* initiator-component
* creation date range

The cleanup process should:

* use dependency-aware deletion order
* use FHIR Transaction Bundles for DELETE operations
* reuse the shared Transaction Bundle Builder
* automatically enforce the globally configured bundle limits
* allow bundle limits to be modified through the Settings UI
* continue processing remaining bundles when failures occur
* provide detailed progress reporting and a final cleanup summary
